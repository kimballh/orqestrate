# Notion Model

This file defines the proposed Notion-side model for artifacts and run history.

## 1. Modeling rules

Use Notion for:

- durable artifacts generated by the harness
- human-readable design, planning, and review documents
- append-only run history and evidence
- summaries that operators can inspect without reading raw logs

Do not use Notion for:

- high-frequency lease renewal
- lock ownership
- worker heartbeat
- authoritative dispatch state

Linear should remain the polling and claim surface.

## 2. Recommended Notion structures

Use two Notion databases:

1. `Issue Artifacts`
2. `Harness Runs`

### 2.1 `Issue Artifacts` database

One page per Linear issue.

Recommended properties:

- `Title`
- `Linear Issue ID`
- `Linear URL`
- `Current Phase`
- `Current Status Snapshot`
- `Latest Run ID`
- `Last Updated At`
- `Review Outcome`
- `Artifact State`

Recommended page body template:

```md
# Context

- Linear issue:
- Problem summary:
- Constraints:

# Design

## Current design summary

## Decisions

## Open questions

# Plan

## Implementation plan

## Risks

## Verification plan

# Implementation Notes

## Change summary

## Files / systems touched

# Review

## Findings

## Rework requested

## Approval notes

# Verification

## Commands / checks

## Evidence

# Decision Log

- timestamped decisions
```

The same page evolves across phases. That keeps context accumulation simple.

### 2.2 `Harness Runs` database

One row per orchestration attempt.

Recommended properties:

- `Run ID`
- `Linear Issue ID`
- `Linear URL`
- `Phase`
- `Status`
- `Owner`
- `Started At`
- `Ended At`
- `Outcome`
- `Artifact Page`
- `Summary`
- `Error`

This database is the run ledger. It should be append-only in normal operation.

## 3. Authority split

| Concern | Authority |
| --- | --- |
| Whether an issue is actionable | Linear |
| Which phase is active | Linear |
| Whether a lease is still valid | Linear |
| Canonical design/plan/review artifact | Notion Issue Artifacts page |
| Run history | Notion Harness Runs database |
| Human-readable evidence | Notion |

Important consequence:

- the orchestrator should poll Linear, not Notion
- the orchestrator should consult Notion after claiming the issue, not before claiming it

## 4. Write ordering

Recommended write order for a phase run:

1. Claim the issue in Linear.
2. Ensure the artifact page exists in Notion.
3. Create a `Harness Runs` row with `Status = claimed` or `running`.
4. Dispatch the worker.
5. Update the Notion artifact page with generated output.
6. Update the `Harness Runs` row with terminal outcome.
7. Update the Linear issue with the next status and machine fields.

That ordering is important:

- claim first so two orchestrators do not both start work
- artifact second so every run has a durable home
- Linear terminal update last so the visible workflow only advances after the artifact exists

## 5. Artifact state recommendations

You can store lightweight derived artifact state in the `Issue Artifacts` page properties, but it should not replace the Linear machine state.

Good artifact properties:

- `Design Ready`
- `Plan Ready`
- `Implementation Notes Present`
- `Review Summary Present`
- `Verification Evidence Present`

These help the orchestrator validate completeness before it moves the Linear issue forward.

## 6. Phase-specific expectations

### Design

Required outputs:

- design summary
- decisions
- open questions
- recommended next step

### Plan

Required outputs:

- implementation plan
- risks
- verification plan
- explicit scope and non-goals

### Implement

Required outputs:

- implementation summary
- touched systems/files
- verification evidence

### Review

Required outputs:

- findings or explicit no-findings statement
- rework summary if needed
- approval summary if approved

### Merge

Required outputs:

- final merge checklist
- residual risks
- release/follow-up notes

## 7. Recommended simplification

For v1, use a single artifact page per issue rather than separate Notion pages for design, plan, implementation, and review.

That gives you:

- a stable artifact URL
- simpler Linear linking
- easier prompt construction
- better context accumulation across sequential phases

You can always split artifacts later if the page gets too large.
